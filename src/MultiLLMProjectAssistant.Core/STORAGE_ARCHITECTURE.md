## 1. Design Philosophy

The storage layer uses a **folder-and-file hierarchy** with JSON serialisation as recommended by the Project Brief (Section 5: "A clean folder-and-file hierarchy is acceptable and encouraged"). This approach was chosen over SQLite because:

- **Transparency:** Project data is human-readable JSON — easy to inspect, debug, and demonstrate.
- **No external dependencies:** The Core project has zero NuGet packages. All persistence uses `System.Text.Json` and `System.IO`.
- **Project isolation:** Each project is a self-contained folder that can be backed up, moved, or deleted independently.
- **Git-friendly:** JSON files produce meaningful diffs when tracked (though `ProjectData/` is excluded from the repository via `.gitignore`).

## 2. Folder Hierarchy

Each project is stored under `{LocalAppData}/MultiLLMProjectAssistant/ProjectData/{ProjectKey}/`:

```
ProjectData/
  {ProjectKey}/                      32-char GUID (no dashes), e.g. "a1b2c3d4e5f6..."
    │
    ├── project.json                 Project metadata (ProjectInfo)
    ├── keys.enc                     Encrypted API keys (binary, AES-256-CBC)
    │
    ├── files/
    │   ├── index.json               List<FileMetadata> — file registry
    │   ├── F001_spec.md             Imported file (ID prefix + original name)
    │   ├── F002_source.cs
    │   └── F003_readme.txt
    │
    ├── memory/
    │   └── items.json               List<MemoryNote> — project memory store
    │
    ├── requests/
    │   ├── REQ-001_request.json     Outgoing LLM request (JSON)
    │   ├── REQ-001_response_raw.json        Raw LLM response
    │   ├── REQ-001_response_normalised.json Normalised response
    │   └── REQ-001_trace.json       Traceability record (TraceRecord)
    │
    └── logs/
        └── app.log                  Structured log (secrets auto-redacted)
```

### 2.1 Path Resolution

The base path defaults to:

```csharp
Path.Combine(
    Environment.GetFolderPath(Environment.SpecialFolder.LocalApplicationData),
    "MultiLLMProjectAssistant",
    "ProjectData"
)
```

- **Windows:** `C:\Users\{user}\AppData\Local\MultiLLMProjectAssistant\ProjectData\`
- **macOS:** `/Users/{user}/.local/share/MultiLLMProjectAssistant/ProjectData/`

The `ProjectService` constructor accepts a custom `basePath` parameter, enabling unit tests to use temporary directories without touching the real file system.

### 2.2 ProjectKey Format

- Generated by `Guid.NewGuid().ToString("N")` — 32 hexadecimal characters, no dashes.
- Example: `a1b2c3d4e5f6a7b8c9d0e1f2a3b4c5d6`
- Used as the folder name and the primary identifier for all operations.

## 4. Service Architecture

### 4.1 Interface Contracts

All services are coded against interfaces, enabling mock-based testing (Moq) and UI-Core decoupling.

| Interface | Implementing Service | Purpose |
|-----------|---------------------|---------|
| `IProjectService` | `ProjectService` | Project CRUD + folder hierarchy |
| `IFileStore` | `FileStoreService` | File import, metadata indexing, content access |
| `IEncryptionService` | `EncryptionService` | AES-256-CBC encrypt/decrypt |
| `IKeyStore` | `KeyStore` | Encrypted API key storage |
| `IProductKeyService` | `ProductKeyService` | Product key validation + feature gating |
| `IAppLogger` | `AppLogger` | Structured logging with redaction |

Additional services without interfaces (used internally):

| Service | Purpose |
|---------|---------|
| `RedactionFilter` | Static — strips API key patterns from strings |
| `FileEmbedder` | Converts file content to LLM-ready text blocks |
| `TraceService` | Per-request traceability records |
| `ExportService` | Sanitised project export |

### 4.3 Data Flow — Full Request Lifecycle

```
User opens project
    │
    ▼
ProjectService.OpenProject(projectKey)
    │ reads project.json, updates LastOpenedAt
    ▼
User attaches files + selects memory mode
    │
    ├──→ FileStoreService.GetFileList(projectKey)     → reads files/index.json
    ├──→ FileEmbedder.EmbedFileContent(projectKey, fileId)  → reads file content
    └──→ [MemoryStore — Kushal's module]              → reads memory/items.json
    │
    ▼
Connector Layer (Cheng) builds JSON request
    │ includes embedded file content + injected memory
    ▼
TraceService.SaveTrace(projectKey, traceRecord)
    │ writes requests/REQ-xxx_trace.json
    ▼
AppLogger.LogRequest(requestId, provider, statusCode)
    │ RedactionFilter.Redact() → writes logs/app.log
    ▼
Response stored in requests/REQ-xxx_response_*.json
    │
    ▼
User may export → ExportService (all content redacted, keys.enc excluded)
```

## 5. Key Design Decisions

| # | Decision | Alternative Considered | Rationale |
|---|----------|----------------------|-----------|
| 1 | **AES-256-CBC + PBKDF2** for encryption | DPAPI (Windows-only) | Cross-platform requirement: Pong develops on macOS. DPAPI requires `net8.0-windows`. |
| 2 | **`net8.0`** target framework | `net8.0-windows` | Core + Tests must build on macOS. WPF-specific code stays in the UI project. |
| 3 | **`System.Text.Json`** for serialisation | `Newtonsoft.Json` | Ships with .NET 8; no external dependency; better performance; source-gen support. |
| 4 | **SHA-256** for file hashing | MD5 (used in UI prototype) | MD5 is cryptographically broken. Design Report requires SHA-256. |
| 5 | **Folder-based storage** with JSON | SQLite | Project Brief recommends file hierarchy. Simpler to implement, inspect, and demonstrate. |
| 6 | **Machine-bound key derivation** | User-supplied master password | Simpler UX (no password prompt). Acceptable for single-user desktop app. |
| 7 | **Sequential File IDs** (F001, F002) | GUID | Human-readable, predictable ordering, easier to debug and demonstrate. |
| 8 | **Compiled Regex** in RedactionFilter | Runtime Regex | `RegexOptions.Compiled` provides ~10x faster matching for repeated calls (logging, export). |
| 9 | **Zero NuGet dependencies** in Core | Various utility packages | Minimises supply chain risk; everything built on BCL (Base Class Library). |
| 10 | **Lock-based thread safety** in AppLogger | `ConcurrentQueue` + background writer | Simpler implementation; sufficient for desktop app with low write frequency. |

## 6. Serialisation Conventions

All JSON serialisation uses `System.Text.Json` with consistent options:

```csharp
private static readonly JsonSerializerOptions JsonOptions = new()
{
    WriteIndented = true
};
```

- **`WriteIndented = true`:** Human-readable JSON for easy inspection during development and demo.
- **Property naming:** Default `PascalCase` (C# convention) — matches the model property names directly.
- **`System.Text.Json` attributes** on models where needed (e.g., `[JsonPropertyName]`).

## 7. Error Handling Strategy

| Scenario | Service | Behaviour |
|----------|---------|-----------|
| Project not found | `ProjectService.OpenProject()` | Throws `DirectoryNotFoundException` |
| File not found for import | `FileStoreService.ImportFileAsync()` | Throws `FileNotFoundException` |
| Binary file content read | `FileStoreService.GetFileContent()` | Throws `NotSupportedException` |
| File exceeds size limit | `FileStoreService.ImportFileAsync()` | Throws `InvalidOperationException` |
| Invalid provider name | `KeyStore.SaveKey()` | Throws `ArgumentException` |
| Empty API key | `KeyStore.SaveKey()` | Throws `ArgumentException` |
| Wrong decryption passphrase | `EncryptionService.Decrypt()` | Throws `CryptographicException` |
| Corrupted ciphertext | `EncryptionService.Decrypt()` | Throws `CryptographicException` |
| Empty/missing `keys.enc` | `KeyStore.LoadKeys()` | Returns empty dictionary (fail-safe) |
| Null input to RedactionFilter | `RedactionFilter.Redact()` | Returns `""` (fail-safe) |
